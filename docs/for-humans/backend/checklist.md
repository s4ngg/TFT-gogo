# checklist.md - 백엔드 검증 기준

> 팀 전체 백엔드 산출물이 올바른지 확인하는 완료 조건이다.
> 도메인별 상세 체크리스트는 각 기능 상세 스펙에서 추가한다.

---

## 공통 API 검증

- [ ] 모든 Controller 응답이 `ResponseEntity<ApiResponse<T>>`를 사용한다.
- [ ] 성공 응답은 `ApiResponse.success(...)`를 사용한다.
- [ ] 실패 경로는 `BusinessException(ErrorCode.XXX)`와 `GlobalExceptionHandler`로 처리한다.
- [ ] Controller에서 raw Map, raw DTO, 문자열 응답을 직접 반환하지 않는다.
- [ ] Request DTO에 필요한 validation이 적용되어 있다.
- [ ] Response DTO는 `from()` 또는 `of()` 정적 팩토리로 생성된다.
- [ ] 페이지 응답은 현재 페이지, 페이지 크기, 전체 수, 전체 페이지 수를 포함한다.
- [ ] API 필드명은 camelCase를 기본으로 한다.

---

## 보안 검증

- [ ] 공개 API와 보호 API가 `SecurityConfig`에서 명확히 구분되어 있다.
- [ ] 인증 없이 접근 가능한 API만 permitAll 처리되어 있다.
- [ ] 공개 허용은 HTTP Method 기준까지 함께 검토되어 있다.
- [ ] 집계 실행, 데이터 생성/수정/삭제 API는 공개 허용되어 있지 않다.
- [ ] `/api/admin/**`는 관리자 인증 없이는 접근할 수 없다.
- [ ] 관리자 토큰 검증 실패 시 401 또는 403 응답이 반환된다.
- [ ] 민감 정보가 응답에 포함되지 않는다.
- [ ] 비밀번호, JWT, 관리자 토큰, Riot API 키가 로그에 남지 않는다.

---

## 예외/로그 검증

- [ ] 존재하지 않는 리소스는 적절한 404 오류로 처리된다.
- [ ] 검증 실패는 400 오류로 처리된다.
- [ ] 인증 실패와 권한 부족이 구분된다.
- [ ] 외부 API 장애가 내부 서버 오류와 구분된다.
- [ ] `System.out.println`이 없다.
- [ ] 운영자가 원인을 추적할 수 있는 로그가 남는다.

---

## Member/Auth 검증

- [ ] 회원가입 validation이 동작한다.
- [ ] 중복 사용자 오류가 명확히 처리된다.
- [ ] 로그인 실패 오류가 명확히 처리된다.
- [ ] 비밀번호가 평문으로 저장되지 않는다.
- [ ] 인증 토큰 또는 세션 정책이 문서화되어 있다.
- [ ] 권한 모델이 관리자 기능 확장을 고려한다.

---

## Summoner/Match 검증

- [ ] Riot ID 기반 소환사 검색이 동작한다.
- [ ] 소환사 기본 정보와 랭크 정보가 응답에 포함된다.
- [ ] 전적 목록이 최신순으로 반환된다.
- [ ] 랭크/일반 게임 유형이 구분된다.
- [ ] 매치 상세에 참가자, 유닛, 아이템, 시너지 정보가 포함된다.
- [ ] Riot API 미제공 데이터는 임의 생성하지 않는다.
- [ ] Riot API 실패와 rate limit이 처리된다.

---

## Deck 검증

- [ ] 메타 덱 목록이 패치 버전과 랭크 필터 기준으로 조회된다.
- [ ] 덱 통계 지표가 응답에 포함된다.
- [ ] 덱 상세에 레벨별 유닛 구성과 추천 배치가 포함된다.
- [ ] 상점 구매 불가 유닛이 공개 응답에서 제외된다.
- [ ] 관리자 숨김 덱은 공개 API에 노출되지 않는다.
- [ ] 관리자 큐레이션 이름과 정렬 우선순위가 공개 API에 반영된다.

---

## Guide 검증

- [ ] 게임가이드 공개 API 큰틀이 프론트 화면 요구사항과 맞다.
- [ ] 가이드 데이터는 유형, 패치 버전, 공개 여부, 정렬 순서를 가질 수 있다.
- [ ] 공개 API는 숨김 데이터를 노출하지 않는다.
- [ ] 관리자 연동이 필요한 데이터와 공개 데이터가 구분되어 있다.

---

## PatchNotes 검증

- [ ] 패치노트 공개 API 큰틀이 프론트 화면 요구사항과 맞다.
- [ ] 패치노트는 버전, 제목, 요약, 적용일, 현재 패치 여부를 가질 수 있다.
- [ ] 변경사항은 카테고리, 변경 타입, 영향도를 가질 수 있다.
- [ ] 공개 API는 숨김 데이터를 노출하지 않는다.
- [ ] 관리자 연동이 필요한 데이터와 공개 데이터가 구분되어 있다.

---

## Community 검증

- [ ] 모집 글 목록 조회가 가능하다.
- [ ] 모집 글 작성 validation이 동작한다.
- [ ] 모집 상태 변경 정책이 명확하다.
- [ ] 검색, 필터, 페이지네이션 요구사항이 정리되어 있다.
- [ ] 채팅 기능의 백엔드 책임 범위가 정리되어 있다.

---

## AI Recommend 검증

- [ ] AI 추천 요청 DTO가 프론트 입력과 맞다.
- [ ] AI 추천 응답 DTO가 프론트 화면과 맞다.
- [ ] ai-server 장애와 타임아웃이 명확히 처리된다.
- [ ] AI 서버 응답이 공통 응답 포맷으로 감싸진다.

---

## 테스트/빌드 검증

- [ ] 신규 Service 로직에 단위 테스트가 있다.
- [ ] 외부 API와 실제 DB에 의존하지 않는 테스트가 우선 작성되어 있다.
- [ ] Entity 변경이 있으면 DB 변경 방식, 적용 순서, 롤백 방법이 문서화되어 있다.
- [ ] `./gradlew test`가 통과한다.
- [ ] `./gradlew build`가 통과한다.
- [ ] Swagger 문서에서 주요 API가 확인된다.

---

## 프론트 연동 검증

- [ ] 프론트 `axiosInstance` baseURL과 백엔드 경로가 중복되지 않는다.
- [ ] 프론트 API 호출부와 백엔드 path, query parameter가 일치한다.
- [ ] 프론트 화면이 기대하는 응답 필드가 백엔드 DTO에 있다.
- [ ] 로딩, 빈 데이터, 오류 상태를 구분할 수 있는 응답이 제공된다.
