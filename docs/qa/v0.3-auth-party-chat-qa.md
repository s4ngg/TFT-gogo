# v0.3 Auth, Party, Realtime Chat QA Plan

작성일: 2026-06-15

## 목적

QA 전에 사용자 담당 범위인 로그인, 회원가입, 소셜 로그인, 커뮤니티탭 파티 만들기/참여, 실시간 채팅의 검증 기준을 하나로 묶는다. 이 문서는 QA 실행 전 체크리스트이며, 실제 QA 결과와 발견 버그는 아래 기록 양식에 따라 별도 이슈 또는 PR comment로 남긴다.

`mergeStateStatus=BLOCKED` 또는 `reviewDecision=REVIEW_REQUIRED`는 사람 리뷰/stacked PR 대기 상태로 기록한다. Critical, 빌드 실패, 보안 위험, 런타임 오류, 사용자 흐름 깨짐이 새로 생기면 QA보다 수정이 우선이다.

## 실행 전 조건

- 통합 QA는 `develop` 단독이 아니라 #375, #376, #377, #358의 head를 합친 로컬 통합 브랜치에서 실행한다.
- QA 통합 브랜치 생성 전 `origin/develop`을 fetch/prune하고, 각 PR head가 최신 원격과 일치하는지 확인한다.
- PR head 병합 중 충돌이 나면 QA를 진행하지 않고 충돌 내용을 별도 기록한다.
- 비밀값, OAuth client secret, GitHub token, Webhook URL, SSH key, `.env` 전체는 기록하지 않는다.
- 서버/API 확인 시 민감 토큰은 마스킹하고, 재현 curl에는 더미 토큰 또는 `<TOKEN>`만 남긴다.

## 공통 통과 기준

- 주요 플로우에서 브라우저 콘솔 `error`, unhandled promise rejection, React render crash가 없다.
- API 오류는 사용자가 이해할 수 있는 메시지로 노출되고, 성공처럼 오인되지 않는다.
- 중복 클릭, Enter 연타, 느린 네트워크에서 중복 요청이나 이중 반영이 없다.
- 모바일 360px, 390px, 768px과 데스크톱에서 버튼/문구/카드/채팅 말풍선이 겹치거나 잘리지 않는다.
- 키보드 `Tab`, `Enter`, `Esc` 흐름에서 포커스가 사라지거나 배경으로 빠지지 않는다.
- `role="status"`, `aria-live`, 폼 오류 연결이 필요한 상태 메시지는 보조기기에서도 추적 가능하다.

## 인증 QA

### 로그인/회원가입

- [ ] 회원가입 성공 시 `/api/v1/auth/signup`이 `ApiResponse<AuthResponse>`로 accessToken과 사용자 정보를 반환한다.
- [ ] 중복 이메일 회원가입은 `EMAIL_ALREADY_EXISTS` 계열로 처리되고, 비밀번호 원문은 응답/로그에 노출되지 않는다.
- [ ] 로그인 성공 시 `/api/v1/auth/login`이 accessToken을 반환하고 이후 요청에 `Authorization: Bearer <TOKEN>`이 붙는다.
- [ ] 없는 이메일과 틀린 비밀번호가 서로 다른 메시지로 계정 존재 여부를 노출하지 않는다.
- [ ] 잘못된 요청 body, 빈 이메일, 짧은 비밀번호는 공통 검증 오류로 처리되고 입력 필드와 오류 메시지가 연결된다.
- [ ] no token, malformed token, expired token, non-numeric JWT subject가 500으로 터지지 않는다.
- [ ] 보호 API는 인증 실패 시 401 계열로 정리되고, 공개 API는 의도된 fallback으로 처리된다.
- [ ] Authorization 헤더가 붙는 요청의 CORS preflight `OPTIONS`가 막히지 않는다.

### `/api/v1/members/me`

- [ ] 유효한 토큰으로 본인 정보만 조회된다.
- [ ] 응답에 `passwordHash`, `socialId`, OAuth provider 내부값, JWT 원문 등 민감/내부 필드가 없다.
- [ ] 토큰은 유효하지만 DB에 회원이 없으면 404 또는 정의된 공통 `ApiResponse` 오류로 내려온다.
- [ ] 새로고침, 새 탭, 브라우저 재진입 후 세션 복원이 동작한다.
- [ ] 401 발생 시 저장된 인증 상태와 `/me` 캐시가 정리되고 사용자가 다시 로그인할 수 있다.
- [ ] 로그아웃 또는 인증 만료 후 이전 사용자 이름/파티/채팅 캐시가 화면에 남지 않는다.

## 소셜 로그인 QA

- [ ] `GET /api/v1/auth/social/{provider}`가 `ApiResponse<SocialLoginStartResponse>`로 절대 authorization URL을 반환한다.
- [ ] 지원 provider는 Google, Naver이며 알 수 없는 provider와 Kakao는 400 계열 오류로 처리된다.
- [ ] 성공 redirect는 `/oauth/callback#accessToken=...`처럼 fragment에 accessToken만 포함한다.
- [ ] 성공 redirect query string에 token, 사용자 정보, provider raw data가 포함되지 않는다.
- [ ] OAuth callback 페이지는 fragment token을 읽은 뒤 URL을 replace해서 token이 주소창에 남지 않게 한다.
- [ ] OAuth callback 후 `/api/v1/members/me`를 다시 조회하고 stale 사용자 캐시를 재사용하지 않는다.
- [ ] 실패 redirect는 raw exception/message가 아니라 `email_exists`, `email_required`, `provider_error` 같은 whitelist code만 전달한다.
- [ ] 일반 계정과 같은 이메일의 소셜 계정은 자동 연결하지 않는다.
- [ ] 소셜 회원의 일반 비밀번호 로그인은 공통 인증 실패로 막힌다.
- [ ] `/api/**`는 stateless JWT 체인을 유지하고 OAuth 브라우저 엔드포인트만 세션을 사용한다.
- [ ] OAuth 관련 설정 누락 시 운영 환경에서 localhost 기본값으로 조용히 성공하지 않는다.

## 파티 모집 QA

### 목록/필터/공개 조회

- [ ] 비로그인 사용자는 파티 목록을 조회할 수 있다.
- [ ] 잘못된 토큰이 있어도 공개 목록 조회는 실패하지 않고 `joined=false`로 내려온다.
- [ ] mode/query 필터가 API 요청과 UI 결과에 일관되게 반영된다.
- [ ] API 실패로 fallback/mock 데이터가 보일 경우 실제 서버 데이터가 아니라는 상태 메시지가 있어야 한다.
- [ ] fallback/mock 데이터에서 참여/취소가 실제 성공처럼 보이면 must-fix로 기록한다.

### 모집글 생성

- [ ] 로그인한 사용자는 제목, 본문, 모드, 모집 인원, 마감 시간, 태그를 입력해 모집글을 만들 수 있다.
- [ ] 제목 필수, 본문 필수, 모집 인원 2-8, 태그 최대 4개/30자, 긴 한글 입력, 과거 마감 시간이 검증된다.
- [ ] 태그는 사용자가 직접 입력한 custom tag가 저장/표시된다.
- [ ] 생성 요청 중 버튼과 Enter 제출이 중복 요청을 만들지 않는다.
- [ ] 생성 실패 시 임시 글이 실제 성공 글처럼 남지 않고 rollback 또는 명확한 실패 상태가 보인다.
- [ ] 생성 성공 응답의 `chatRoomId`가 해당 파티 전용 채팅방과 연결된다.

### 참여/취소

- [ ] 비로그인 사용자의 생성/참여/취소는 성공처럼 보이지 않고 로그인 필요 상태로 정리된다.
- [ ] 참여 버튼 클릭 시 바로 참여 처리되며 승인 대기 상태로 가지 않는다.
- [ ] 작성자는 `joined=true`로 보이지만 일반 참여자처럼 취소 버튼으로 빠질 수 없다.
- [ ] 중복 join은 idempotent이며 인원수가 두 번 증가하지 않는다.
- [ ] 정원 초과, 마감 지난 글, closed 글은 join이 409 계열로 막히고 `currentMembers > maxMembers`가 되지 않는다.
- [ ] 참여/취소 API 실패 시 UI 인원수와 버튼 상태가 rollback된다.
- [ ] 더블클릭, 빠른 다른 글 전환, 느린 네트워크에서 참여 상태가 꼬이지 않는다.
- [ ] 여러 사용자가 거의 동시에 join해도 정원 초과와 중복 application이 생기지 않는다.

## 실시간 채팅 QA

### 계약/보안

- [ ] 채팅 REST/SSE는 인증 없이 접근할 수 없다.
- [ ] 파티 참여자만 채팅 가능 여부는 MVP 후속 범위라면 known risk/out-of-scope로 명시한다.
- [ ] 요청 body는 `roomId`, `content`만 받으며 `senderName`, `tier`, `userId` 같은 사칭 필드는 클라이언트에서 보낼 수 없거나 서버에서 무시/거부된다.
- [ ] senderName은 인증된 사용자 닉네임에서 서버가 만든다.
- [ ] token query string 없이 Authorization 헤더 기반으로 SSE를 구독한다.

### 메시지 전송/조회

- [ ] blank, trim 후 빈 문자열, 500자 초과 메시지는 공통 400 응답으로 처리된다.
- [ ] 존재하지 않는 roomId, 임의 조작 roomId, 잘못된 형식은 500이 아니라 400/404 계열로 정리된다.
- [ ] 전송 성공 시 REST 응답과 SSE echo가 중복 표시되지 않는다.
- [ ] 전송 실패 메시지는 유실되지 않고 재시도 가능하거나 실패 상태가 보인다.
- [ ] 메시지 순서, 시간 표시, 긴 메시지 줄바꿈, 긴 닉네임 표시가 깨지지 않는다.
- [ ] 잘못된 createdAt 값이 들어와도 채팅 패널 전체 렌더가 깨지지 않는다.

### SSE 연결

- [ ] SSE 응답은 `text/event-stream`이고 최초 snapshot 이벤트가 온다.
- [ ] 이후 message 이벤트가 같은 room cache에 반영된다.
- [ ] snapshot은 이전 캐시와 무조건 merge하지 않고 서버 기준 상태로 정리된다.
- [ ] 파티방 전환, 화면 이탈, 로그아웃, 재연결 시 이전 SSE가 닫힌다.
- [ ] 재연결 후 메시지 중복 수신이 없다.
- [ ] 서버 재시작/네트워크 끊김/stream 종료 시 연결 상태가 사용자에게 표시된다.

## 접근성/상태 QA

- [ ] 로그인/회원가입/소셜 로그인 버튼은 키보드로 접근하고 실행할 수 있다.
- [ ] 파티 생성 폼의 모든 입력은 label 또는 aria label을 가진다.
- [ ] 입력 오류는 시각적 메시지만이 아니라 해당 input과 연결된다.
- [ ] "모집글 작성" 클릭 후 제목 input으로 포커스가 이동한다.
- [ ] 파티 참여/취소, 채팅 전송 pending 상태는 disabled 속성과 상태 문구가 함께 반영된다.
- [ ] 401 리다이렉트, 404 화면, 오류 토스트, 재연결 실패 이후 포커스 위치가 예측 가능하다.
- [ ] 채팅 새 메시지/연결 상태/전송 실패 상태는 `role="status"` 또는 `aria-live`로 확인 가능하다.
- [ ] 모바일에서 파티 카드, 작성 폼, 채팅 입력창, 버튼 텍스트가 겹치거나 잘리지 않는다.

## Swagger/API 확인

- [ ] `/v3/api-docs`에서 Auth, Member, CommunityParty, Chat API가 노출된다.
- [ ] REST API는 `ResponseEntity<ApiResponse<T>>` 계약으로 문서화되어 있다.
- [ ] SSE는 `ApiResponse<T>`가 아닌 stream 예외 계약으로 문서화되어 있다.
- [ ] BearerAuth가 보호 API에 표시된다.
- [ ] principal/userId 같은 내부 인증 파라미터가 request body로 노출되지 않는다.
- [ ] PR 본문과 Discord status에 Swagger/API 확인 여부를 기록한다.

## Known Risk / 후속 범위

- Kakao 로그인은 `account_email` 권한이 없어 현재 QA 범위에서 제외하며, 권한 확보 후 별도 이슈로 재검토한다.
- #358 MVP 채팅은 메모리 저장이므로 서버 재시작 또는 멀티 인스턴스에서 메시지가 유지/공유되지 않는다.
- 파티 참여자만 채팅 가능하도록 제한하는 멤버십 검증은 후속 slice로 남아 있다면 QA 결과에 명시한다.
- `chat_rooms` DB row 영속화와 Redis fan-out은 후속 범위다.
- owner close/delete 정책은 아직 별도 정의가 필요하다.
