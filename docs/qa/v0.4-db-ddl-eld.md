# v0.4 DB DDL/ELD QA 기준

## 목적

QA 전에 새 MySQL DB에서 백엔드가 최소 기능 테스트 가능한 상태로 기동되도록 `local-smoke` DDL과 시드를 정리한다.

- 대상 DB: MySQL 8.0
- 기준 코드: 현재 JPA Entity + `docs/for-ai/spec`
- DDL 파일: `backend/src/main/resources/db/local-smoke/01_schema.sql`
- Seed 파일: `backend/src/main/resources/db/local-smoke/02_seed.sql`
- ERD Cloud 반영 체크리스트: `docs/qa/v0.4-erd-cloud-sync-checklist.md`

## 적용 방법

### 새 Docker 볼륨에서 자동 적용

`docker-compose.yml`은 MySQL 빈 볼륨 생성 시 `/docker-entrypoint-initdb.d`로 `01_schema.sql`, `02_seed.sql`을 자동 실행한다.

```powershell
docker compose up -d mysql redis
docker compose up --build backend
```

주의: MySQL Docker init SQL은 `mysql-data` named volume이 처음 만들어질 때만 실행된다.

## 실행 검증 이력

2026-06-16 기준 MySQL 8.0.44 임시 DB에 `01_schema.sql`, `02_seed.sql`을 순서대로 적용해 확인했다.

| Check | Result |
| --- | --- |
| `01_schema.sql` 적용 | Pass |
| `02_seed.sql` 적용 | Pass |
| 생성 테이블 수 | 19개 |
| `chat_rooms` seed | 4개 (`general`, `deck-guide`, `party-recruitment`, `question-answer`) |
| `guides` seed | 4개 |
| `patch_notes` seed | 1개 |
| `patch_note_changes` seed | 2개 |
| `backend` `compileJava` | Pass |

검증은 기존 로컬 `tftgogo` DB를 변경하지 않고 별도 임시 DB에서 수행했다. Docker daemon이 꺼져 있거나 기존 볼륨이 남아 있어도, 새 DB 기준 SQL 자체는 적용 가능함을 확인한 상태다.

### 기존 볼륨을 초기화하고 다시 적용

로컬 QA DB 데이터를 지워도 되는 경우에만 사용한다.

```powershell
docker compose down -v
docker compose up -d mysql redis
docker compose up --build backend
```

### 새/임시 QA DB에 수동 적용

아래 명령은 새로 만든 QA DB나 폐기 가능한 임시 DB에만 실행한다. `01_schema.sql`은 새 DB 부트스트랩용 DDL이고, `02_seed.sql`은 `patch_notes.is_current` 값을 조정하며 local-smoke patch change row를 삭제/재삽입한다. 운영/공유/기존 개발 DB에 데이터를 보존한 채 적용하는 절차가 아니므로, 기존 DB 구조가 불완전하면 새 DB로 다시 만들거나 별도 ALTER/seed 스크립트를 준비해야 한다.

```powershell
Get-Content -Encoding UTF8 backend/src/main/resources/db/local-smoke/01_schema.sql |
  docker exec -i tftgogo-mysql mysql --default-character-set=utf8mb4 -utftuser -ptftpass tftgogo

Get-Content -Encoding UTF8 backend/src/main/resources/db/local-smoke/02_seed.sql |
  docker exec -i tftgogo-mysql mysql --default-character-set=utf8mb4 -utftuser -ptftpass tftgogo
```

| `chat_messages` | `id` | `BIGINT` | PK, auto increment | 비로컬 런타임 메시지 식별자. local in-memory `ChatMessage.id`는 UUID 문자열이다. |

| `chat_messages` | `content` | `TEXT` | NOT NULL | 메시지 내용. 현재 API는 500자 이하로 검증한다. |

채팅 MVP는 local 프로필에서만 `InMemoryChatServiceImpl`이 메모리에 메시지를 저장한다. 비로컬 런타임은 `chat_rooms`, `chat_messages`를 사용해 최근 메시지를 영속화하므로 공유/배포 QA에서는 DB row와 API 응답을 함께 확인할 수 있다.

현재 채팅 MVP의 유효 room id는 코드 상수 기준으로만 제한된다.

- `general`
- `deck-guide`
- `party-recruitment`
- `question-answer`

`02_seed.sql`은 위 네 room id를 `chat_rooms`에 upsert한다. 비로컬 런타임 QA에서는 DB row와 현재 API 동작을 함께 확인한다.

## 기능별 QA 가능 범위

| 기능 | DB 기준 | QA 메모 |
| --- | --- | --- |
| 회원가입 | `users` | 이메일/닉네임 중복, BCrypt password hash 저장 확인 가능 |
| 로그인 | `users` | 일반 계정 로그인 가능 |
| 내 정보 조회 | `users` | Bearer access token 필요 |
| 소셜 로그인 시작 | OAuth2 client registration | DB만으로 E2E 완료 불가. provider client-id/client-secret과 callback URL 설정 필요 |
| 소셜 회원 저장 | `users.social_provider`, `users.social_id` | 실제 provider 콜백 성공 후 저장 가능 |
| 파티 모집글 생성 | `party_posts`, `party_post_tags` | 작성자는 `current_members = 1`, 응답 `chatRoomId = party-recruitment` |
| 파티 즉시 참여 | `party_applications` | `status = ACCEPTED`, `(party_post_id, user_id)` unique |
| 실시간 채팅 조회/SSE | `chat_messages` / local은 in-memory | GET messages, GET stream은 비로그인 허용 |
| 실시간 채팅 전송 | `users` + `chat_messages` / local은 in-memory | POST message는 Bearer access token 필요 |

## Swagger/API 확인 항목

PR 검증 시 아래를 확인하고 PR 본문에 결과를 적는다.

- `/v3/api-docs`가 200으로 응답한다.
- Auth/Member/Party/Chat API가 Swagger에 노출된다.
- Controller 응답 계약은 `ApiResponse<T>` envelope를 유지한다.
- Chat GET messages/SSE는 공개 접근, POST message는 Bearer 인증 필요로 문서/코드가 일치한다.
- Party create response의 `chatRoomId`는 고정 `party-recruitment`이며 파티별 새 room tab을 만들지 않는다.
- 소셜 로그인은 provider secret 없이 E2E 성공으로 표기하지 않는다.

## ERD Snapshot과 다른 점

- `chat_rooms`, `chat_messages`는 ERD에 있으나 현재 JPA Entity가 없어 예약 테이블로만 둔다.
- 고정 커뮤니티 채팅 room id 관리를 위해 `chat_rooms.room_key` unique 컬럼을 추가했다.
- `party_posts.tier` 전용 컬럼은 최종 ERD에 없으므로 만들지 않는다. 티어 조건은 `party_post_tags.tag`에 커스텀 태그로 저장한다.
- `party_post_tags`는 ERD에는 없지만, 사용자가 직접 입력하는 파티 태그와 티어 조건을 저장하기 위한 보조 테이블이다.
- 파티 생성 응답은 파티 전용 room이 아니라 고정 `party-recruitment` room을 반환한다.
