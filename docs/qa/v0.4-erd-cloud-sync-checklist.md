# v0.4 ERD Cloud 동기화 체크리스트

## 목적

강사님 QA 전에 새 DB에서 기능 테스트가 막히지 않도록 `local-smoke` DDL을 먼저 기준선으로 확정하고, 같은 불일치가 반복되지 않도록 ERD Cloud와 팀 작업 규칙을 맞춘다.

## 진행 순서

1. QA용 DDL 기준선 확정
2. ERD Cloud를 DDL/ELD 기준으로 수정
3. 이후 테이블/컬럼 변경 시 팀 협의 규칙 적용

## 1. QA용 DDL 기준선

기준 파일은 아래 두 개다.

- `backend/src/main/resources/db/local-smoke/01_schema.sql`
- `backend/src/main/resources/db/local-smoke/02_seed.sql`

검증 기준은 `docs/qa/v0.4-db-ddl-eld.md`에 기록된 MySQL 8.0.44 임시 DB 적용 결과를 따른다.

현재 새 DB 부트스트랩 기준:

- 생성 테이블: 19개
- 고정 채팅 room seed: 4개
- Guide seed: 4개
- PatchNotes seed: 1개
- PatchChanges seed: 2개

## 2. ERD Cloud 우선 수정 항목

아래 항목은 QA 전에 ERD Cloud와 코드/DDL 기준이 어긋나지 않도록 우선 맞춘다.

| P0 | `patch_changes` 테이블 추가 또는 기존 `patch_note_changes`를 `patch_changes`로 정렬 | JPA Entity/DDL은 `patch_changes`를 사용 |

| P0 | 타입 차이 정렬 | `ENUM` -> `VARCHAR`, `DECIMAL` -> `DOUBLE`, `patch_notes.summary TEXT` -> `VARCHAR(500)`, boolean 계열 -> `TINYINT(1)` |
| P1 | `cached_summoner`, `cached_rank`, `cached_match`, `cached_match_participant` 추가 | Summoner/Match cache JPA 테이블이 snapshot에 없음 |
| P1 | `deck_curations` 추가 | Admin deck curation JPA 테이블이 snapshot에 없음 |

### Chat

현재 채팅 MVP는 DB 영속화가 아니라 `InMemoryChatServiceImpl` 기준으로 동작한다. ERD Cloud에는 후속 영속화용 예약 테이블로 표시한다.

| Table | ERD Cloud 반영 기준 |
| --- | --- |
| `chat_rooms` | ELD 예약 테이블. 현재 런타임 API는 읽거나 쓰지 않음 |
| `chat_rooms` | `room_key` VARCHAR(80) NOT NULL unique 추가 |
| `chat_rooms` | `creator_id`는 nullable FK -> `users.user_id` |
| `chat_rooms` | `party_post_id`는 nullable FK -> `party_posts.id` |
| `chat_rooms` | 고정 room key는 `general`, `deck-guide`, `party-recruitment`, `question-answer` |
| `chat_messages` | ELD 예약 테이블. 현재 메시지는 in-memory 저장 |
| `chat_messages` | `user_id` FK -> `users.user_id`, `chat_room_id` FK -> `chat_rooms.id` |

중요: 파티 모집글 생성은 파티 전용 채팅방을 만들지 않고 고정 `party-recruitment` room id를 반환한다. 일반 유저가 새 채팅 탭/room을 만드는 기능은 MVP 범위 밖이다.

## 3. 팀 협의 규칙 제안

앞으로 같은 문제가 반복되지 않도록 아래 규칙을 팀에 공유한다.

- DB 기준선은 `01_schema.sql`과 ERD Cloud를 동시에 맞춘다.
- Entity/Repository/DTO 변경으로 테이블 또는 컬럼이 바뀌면 PR 본문에 DB 영향 여부를 반드시 적는다.
- 새 테이블, 새 컬럼, FK, unique, check constraint가 생기면 ERD Cloud 반영 여부를 PR 체크리스트에 포함한다.
- `ddl-auto=none` 환경이므로 Entity 수정만으로 DB가 바뀌지 않는다는 점을 PR 리뷰에서 확인한다.
- QA DB는 기존 개발 DB에 덮어쓰지 말고 새 DB 또는 폐기 가능한 임시 DB 기준으로 부트스트랩한다.
- `02_seed.sql`은 `patch_notes.is_current`와 local-smoke 데이터를 조정하므로 운영/공유 DB에 데이터 보존 절차처럼 직접 실행하지 않는다.
- 소셜 로그인 provider `client-secret`, JWT secret, DB password 같은 비밀값은 ERD Cloud, PR, Discord 공지에 쓰지 않는다.
